

# 异常日志持久化方案（ClickHouse + Redis）

## 一、整体思路

### 预测产出
模型输出异常 BlockId 及其 29 维特征（E1~E29）和异常分数（如概率）。

### 存储分层
- **全量存储**：ClickHouse（列式分析数据库），用于归档、分页查询和复杂筛选。
- **热点缓存**：Redis 有序集合（Sorted Set）存储 Top N 异常，Redis Hash 存储异常详情。

### 查询接口
- **Top N 异常**：直接从 Redis 读取，毫秒级响应。
- **分页/条件查询**：从 ClickHouse 查询，支持高效筛选和排序。

## 二、具体实现方案

### 1. ClickHouse 表设计

```sql
-- 使用 ReplacingMergeTree 引擎，支持按 block_id 去重（保留最新插入的行）
CREATE TABLE anomaly_blocks (
    block_id String,
    E1 Int32,
    E2 Int32,
    E3 Int32,
    E4 Int32,
    E5 Int32,
    E6 Int32,
    E7 Int32,
    E8 Int32,
    E9 Int32,
    E10 Int32,
    E11 Int32,
    E12 Int32,
    E13 Int32,
    E14 Int32,
    E15 Int32,
    E16 Int32,
    E17 Int32,
    E18 Int32,
    E19 Int32,
    E20 Int32,
    E21 Int32,
    E22 Int32,
    E23 Int32,
    E24 Int32,
    E25 Int32,
    E26 Int32,
    E27 Int32,
    E28 Int32,
    E29 Int32,
    anomaly_score Float32,
    detected_at DateTime DEFAULT now(),
    batch_id String
) ENGINE = ReplacingMergeTree(detected_at)   -- 以 detected_at 为版本，保留最新行
ORDER BY (block_id);                         -- 排序键（也是去重键）
```

**说明**：
- ClickHouse 不支持单行更新，通过 `ReplacingMergeTree` 在后台合并时保留最新版本（`detected_at` 最大）。
- 查询时需要使用 `FINAL` 关键字或 `GROUP BY block_id` 获取唯一最新记录。
- 为加速按异常分数排序的查询，可添加跳数索引：
  ```sql
  ALTER TABLE anomaly_blocks ADD INDEX idx_score anomaly_score TYPE minmax GRANULARITY 1;
  ALTER TABLE anomaly_blocks MATERIALIZE INDEX idx_score;
  ```

### 2. Redis 数据结构

#### Top N 异常列表（例如 N=10）
- **Key**: `anomaly:top`
- **类型**: Sorted Set
- **Member**: `block_id`
- **Score**: `anomaly_score`

```bash
ZADD anomaly:top 0.95 "blk_123"
ZREMRANGEBYRANK anomaly:top 0 -11   # 保留前10
ZREVRANGE anomaly:top 0 9 WITHSCORES
```

#### 异常详情缓存（可选，加速详情展示）
- **Key**: `anomaly:detail:{block_id}`
- **类型**: Hash
- **字段**: `E1`, `E2`, ..., `E29`, `anomaly_score`

```bash
HSET anomaly:detail:blk_123 E1 2 E2 0 ... anomaly_score 0.95
HGETALL anomaly:detail:blk_123
```

### 3. 存储流程（每次发现新异常时）

```python
import clickhouse_connect
import redis
import pandas as pd
from datetime import datetime

client = clickhouse_connect.get_client(host='localhost', port=9000, username='default', password='')
r = redis.Redis(host='localhost', port=6379, db=0)

def save_anomaly(block_id, features, anomaly_score, batch_id=None):
    # features = [E1, E2, ..., E29] （长度为29的列表）
    
    # 1. 写入 ClickHouse（直接插入新行）
    data = [{
        'block_id': block_id,
        'E1': features[0], 'E2': features[1], 'E3': features[2],
        'E4': features[3], 'E5': features[4], 'E6': features[5],
        'E7': features[6], 'E8': features[7], 'E9': features[8],
        'E10': features[9], 'E11': features[10], 'E12': features[11],
        'E13': features[12], 'E14': features[13], 'E15': features[14],
        'E16': features[15], 'E17': features[16], 'E18': features[17],
        'E19': features[18], 'E20': features[19], 'E21': features[20],
        'E22': features[21], 'E23': features[22], 'E24': features[23],
        'E25': features[24], 'E26': features[25], 'E27': features[26],
        'E28': features[27], 'E29': features[28],
        'anomaly_score': anomaly_score,
        'detected_at': datetime.now(),
        'batch_id': batch_id or ''
    }]
    client.insert_df('anomaly_blocks', pd.DataFrame(data))
    
    # 2. 更新 Redis Top N
    r.zadd('anomaly:top', {block_id: anomaly_score})
    r.zremrangebyrank('anomaly:top', 0, -11)  # 保留前10
    
    # 3. 缓存详情（只缓存 Top N 的详情以节省内存）
    if r.zrank('anomaly:top', block_id) is not None:
        feature_dict = {f'E{i}': features[i-1] for i in range(1,30)}
        feature_dict['anomaly_score'] = anomaly_score
        r.hset(f'anomaly:detail:{block_id}', mapping=feature_dict)
```

### 5. 查询接口实现

#### 获取 Top N 异常
```python
top = r.zrevrange('anomaly:top', 0, 9, withscores=True)
# 返回 [(block_id, score), ...]
```

#### 获取异常详情（先查 Redis，未命中则查 ClickHouse）
```python
def get_anomaly_detail(block_id):
    detail = r.hgetall(f'anomaly:detail:{block_id}')
    if detail:
        # 将 bytes 转换为字符串/数值
        return {k.decode(): v.decode() if isinstance(v, bytes) else v for k, v in detail.items()}
    
    # 从 ClickHouse 查询（使用 FINAL 获取最新版本）
    result = client.query_df(f"""
        SELECT block_id, E1, E2, E3, E4, E5, E6, E7, E8, E9,
               E10, E11, E12, E13, E14, E15, E16, E17, E18, E19,
               E20, E21, E22, E23, E24, E25, E26, E27, E28, E29,
               anomaly_score
        FROM anomaly_blocks FINAL
        WHERE block_id = '{block_id}'
    """)
    if not result.empty:
        return result.iloc[0].to_dict()
    return None
```

## 三、与“向量检索”的对比及结论

- **向量检索适用场景**：非结构化数据（文本、图像）的语义相似搜索。你的 E1~E29 是结构化数值，直接精确查询或范围查询即可，无需向量化。
- **普通 Redis 方案优势**：实现简单、资源占用低、查询性能极高（毫秒级），完全满足 Top N 和详情查询需求。
- **最终建议**：采用上述“ClickHouse + 普通 Redis”方案，避免过度设计。

---

## 四、补充说明

- **异常分数定义**：可使用模型输出的异常概率（0~1），也可自定义加权公式（如 `anomaly_score = 0.5*E4 + 0.3*E7 + 0.2*E23`）。
- **Redis 内存优化**：如果异常总数巨大，可只缓存 Top N 的详情，其余仅存 ClickHouse。
- **数据一致性**：先写 ClickHouse，后更新 Redis；若 Redis 更新失败，下次查询时从 ClickHouse 重建即可。
- **ClickHouse 更新机制**：如需更正某个 block_id 的异常分数，只需插入一条新记录（detected_at 为当前时间），查询时通过 `FINAL` 或 `GROUP BY` 自动取最新值。
- **批处理优化**：高并发场景下，建议批量插入 ClickHouse（如每 1000 条一次），减少网络开销。

通过上述方法，你可以高效地存储、缓存和查询异常日志，满足实时展示 Top 异常以及分页浏览的需求。
