## 今日对话总结：你的当前任务与实现路径

经过今天的讨论，你已明确要将**离线批处理**与**在线实时分析**分离，并采用 **ClickHouse + Redis** 的架构来存储、统计和缓存异常日志。最终目标是：当新日志流入时，自动更新统计、预测异常、缓存热点数据，并支持动态查询。

### 一、你已经确定的核心架构

1. **数据源与传输**  
   - 离线批量上传：发送到 Kafka Topic `hdfs-logs-offline`，每条消息携带唯一 `batch_id`（如文件上传时间戳）。  
   - 在线实时流：发送到 Kafka Topic `hdfs-logs-online`，消息不附加批次标识。

2. **ClickHouse 职责**  
   - 使用 `Kafka` 引擎表消费两个 Topic，通过物化视图将原始日志写入 `hdfs_logs` 表。  
   - 从 `hdfs_logs` 实时统计每个 BlockId 的 E1~E29 事件频次，存入 `block_event_stats`（原始统计表）。  
   - 预测脚本从 `block_event_stats` 读取特征，调用 MLP 模型，将判定为异常的 BlockId 及其特征写入 **异常结果表 `anomaly_blocks`**（与原始统计表分离）。  
   - `anomaly_blocks` 表使用 **TTL + 分区**（如按天分区，保留最近 7 天），自动滑动窗口。

3. **Redis 职责**  
   - 定期（如每 10 秒）从 `anomaly_blocks` 增量查询新异常（基于 `detected_at` 字段）。  
   - 使用 **Sorted Set** 维护异常分数最高的 Top 10 BlockId，使用 **Hash** 缓存每个异常 BlockId 的详情（E1~E29 + 异常分数）。  
   - 同步时覆盖旧值，保证缓存始终是最新状态。

4. **查询接口**  
   - Top 10 异常：直接读 Redis Sorted Set。  
   - 分页/条件查询：从 ClickHouse `anomaly_blocks` 查询（支持按异常分数排序、按 E1~E29 筛选）。  
   - CSV 归档：定期（如每天）从 `anomaly_blocks` 导出备份，不依赖 CSV 作为实时数据源。

### 二、你当前需要具体执行的步骤

#### ✅ 第一步：改造 ClickHouse 表结构

1. **创建异常结果表 `anomaly_blocks`**（带 TTL 和分区）  
   ```sql
   CREATE TABLE anomaly_blocks (
       block_id String,
       E1 Int32, E2 Int32, ..., E29 Int32,
       anomaly_score Float32,
       detected_at DateTime64(3) DEFAULT now(),
       batch_id String
   ) ENGINE = ReplacingMergeTree(detected_at)
   PARTITION BY toYYYYMMDD(detected_at)
   ORDER BY block_id
   TTL detected_at + INTERVAL 7 DAY DELETE;
   ```

2. **确保原始统计表 `block_event_stats` 已存在**（包含 `last_updated` 字段，用于增量查询特征）。

#### ✅ 第二步：编写预测脚本（定期从 `block_event_stats` 读取特征 → 调用模型 → 写入 `anomaly_blocks`）

last_predict_time = get_last_predict_time()   # 记录上次预测的时间戳
query = f"""
SELECT block_id, E1, E2, ..., E29, max(last_updated) as last_updated
FROM block_event_stats
WHERE last_updated > '{last_predict_time}'

df = client.query_df(query)
for _, row in df.iterrows():

    score = model.predict_proba([features])[0][1]   # 异常概率

#### ✅ 第三步：实现 Redis 增量同步脚本（从 `anomaly_blocks` 拉取新异常）

```python
last_sync = redis.get('anomaly:last_sync_time') or '1970-01-01'
query = f"""
SELECT block_id, E1, E2, ..., E29, anomaly_score, max(detected_at) as detected_at
FROM anomaly_blocks
WHERE detected_at > '{last_sync}'
GROUP BY block_id
"""
df = client.query_df(query)
if not df.empty:
    pipe = redis.pipeline()
    for _, row in df.iterrows():
        pipe.zadd('anomaly:top', {row['block_id']: row['anomaly_score']})
        pipe.hset(f'anomaly:detail:{row["block_id"]}', mapping={f'E{i}': row[f'E{i}'] for i in range(1,30)} | {'anomaly_score': row['anomaly_score']})
    pipe.zremrangebyrank('anomaly:top', 0, -11)
    pipe.execute()
    redis.set('anomaly:last_sync_time', df['detected_at'].max())
```

#### ✅ 第四步：设置定时任务

- 预测脚本：每 30 秒执行一次（或与 Kafka 消费速度匹配）。  
- Redis 同步脚本：每 10 秒执行一次（可根据异常产生频率调整）。  
- 使用 `cron` 或 `systemd timer`，或简单的 `while true` 循环。

#### ✅ 第五步：提供查询接口

- **获取 Top 10 异常**：`redis.zrevrange('anomaly:top', 0, 9, withscores=True)`  
- **获取异常详情**：优先查 Redis Hash，未命中则查 ClickHouse。  
- **分页查询全量异常**：执行 ClickHouse SQL（如 `SELECT ... FROM anomaly_blocks FINAL ORDER BY anomaly_score DESC LIMIT 20 OFFSET 0`）。

#### ✅ 第六步：CSV 归档（可选）

每天凌晨导出 `anomaly_blocks` 当前数据：
```bash

```

### 三、注意事项

- **TTL 自动删除旧数据**：无需手动清理，但需确认 ClickHouse 版本支持（>= 20.3）。  
- **Redis 内存**：如果异常总量很大，可只缓存 Top N 的详情（如上代码已实现）。  
- **数据一致性**：先写 ClickHouse，后更新 Redis；若 Redis 更新失败，下次同步会重复处理（幂等）。  
- **性能优化**：批量插入 ClickHouse（如每 1000 条一次），Redis 使用 pipeline。
